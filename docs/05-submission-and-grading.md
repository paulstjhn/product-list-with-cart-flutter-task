# Evaluation & Submission

## What to Submit

- GitHub repository link — public, or shared with your mentor
- A short `README.md` — what tier you attempted, how to run the project, any known issues
- A demo recording (screen record) **or** an APK file for Android testing

---

## How It Will Be Reviewed

Submissions will be reviewed on three dimensions:

| Dimension | What we're looking at |
|-----------|----------------------|
| **Does it work?** | Run the app, test every feature, look for edge cases (empty cart, rapid taps, etc.) |
| **Is the code sound?** | Code review via GitHub — BLoC correctness, widget decomposition, naming, structure |
| **Can you explain it?** | Brief verbal walkthrough — mentee explains one decision they made and why |

The verbal walkthrough is not a formality. Pick a decision you genuinely had to think about — why you structured the repository the way you did, why you chose a particular widget boundary, how you handled a tricky edge case — and be ready to explain it. If you can't explain a decision in your own project, that's a gap worth closing before submission.

---

## Grade Boundaries

| Grade | Score | What it means |
|-------|-------|----------------|
| **Distinction** | 85 – 100 | All features work, BLoC is clean, code is well-structured and readable |
| **Merit** | 70 – 84 | Most features work, minor BLoC issues, structure is mostly correct |
| **Pass** | 50 – 69 | Core features work but structural or BLoC gaps are significant |
| **Needs Work** | < 50 | Major features missing or app does not run — revision required |

## Before You Submit

- [ ] Re-read [`06-ground-rules.md`](06-ground-rules.md) — a ground rule violation can cap your grade regardless of feature completeness
- [ ] Run through [`../CHECKLIST.md`](../CHECKLIST.md) in full
- [ ] Confirm the app runs from a clean clone — not just on your machine with leftover state
- [ ] Test edge cases yourself before your mentor does: empty cart, rapid repeated taps, incrementing/decrementing at the boundaries
